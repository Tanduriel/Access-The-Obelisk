## Game Key Bindings

Primary input classes:

- `ActionsController`: Unity Input System action asset wrapper.
- `InputController`: central runtime input handler.

Player actions:

- Movement: arrow keys, WASD, gamepad dpad, gamepad left stick.
- Fire/activate: gamepad south button, touch tap, mouse left click through UI raycast. With keyboard shortcuts enabled, left/right Ctrl calls the same activate path.
- Escape: Escape key.
- Combat end turn: Space calls `MatchManager.KeyboardSpace()`, which calls `MatchManager.EndTurn()` when keyboard shortcuts are enabled, it is the player's turn, combat is not busy, no blocking combat windows are active, and the end-turn button is visible.
- Mod fallback note: `CombatHandler` also calls `MatchManager.KeyboardSpace()` on Space while combat is active. This covers cases where the game's Input System `KeyBinding` action is not firing, without bypassing the game's own end-turn conditions.
- KeyBinding action: digits 0-9, F1-F12, Tab, Space, R, E, S, A, W, Q, Enter, Ctrl, Alt, PageDown, PageUp, gamepad triggers/shoulders/buttons.
- Zoom/scroll: mouse scroll and gamepad vectors, mainly for Tome and CardCraft screens.

UI actions:

- Navigate: arrow keys, WASD, gamepad sticks, gamepad dpad.
- Submit: generic submit binding.
- Cancel: generic cancel binding.
- Point/click/scroll/middle/right click: mouse, pen, touch.

Important behavior:

- `InputController.DoMovementVector()` routes movement to the currently active manager, including `KeyboardManager`, `AlertManager`, `SettingsManager`, `CardScreenManager`, `DamageMeterManager`, `TomeManager`, `MainMenuManager`, `PerkTree`, `SandboxManager`, `MadnessManager`, `LobbyManager`, `HeroSelectionManager`, `GiveManager`, `IntroNewGameManager`, `CinematicManager`, `CardCraftManager`, `CardPlayerManager`, `EventManager`, `TownManager`, `MapManager`, `RewardsManager`, `LootManager`, `FinishRunManager`, and `MatchManager`.
- This means the accessibility mod should initially announce the game's existing focus/navigation rather than replacing navigation.
- `GameManager.Instance.EscapeFunction()` is the central Escape handler.
- `GameManager.Instance.ConfigKeyboardShortcuts` gates many keyboard shortcuts in `DoKeyBinding()`.
- F1-F12 and period/comma are developer-mode actions in `InputController`; avoid them as single-key mod controls.

## Safe Mod Keys

Tentative safe approach:

- Prefer no new single-key gameplay controls at first.
- Reuse existing game navigation and announce focus/state changes.
- For mod-only diagnostics, use modifier combinations such as Ctrl+Shift+F9 only after in-game verification.
- Plain `F` is currently not present in the game's documented `KeyBinding` action and was not found in `InputController` key handling. It is used by the accessibility mod to open the native deck view/current combat draw pile. `Shift+F` opens the current combat discard pile. `Ctrl+Shift+F` opens the current combat vanished-card pile through the game's native `ShowCharacterWindow("combatvanish")` path.
- `Ctrl+E` is used by `CombatHandler` to announce current combat energy on demand. Plain `E` remains avoided because it is a combat emote key.
- `I` is used by `CharacterInfoHandler` to open the game's native character information window where available. Plain `I` was not found in `InputController` key handling.
- `B` is used by `TomeHandler` to open the game's native Tome of Knowledge where available. `B` was not found in `InputController` key handling. `Ctrl+T` is now reserved by `CombatHandler` to announce combat turn order.
- `Ctrl+H` is used by `CombatHandler` to announce HP for the focused combat character, or the active-turn character if no character is focused. `Ctrl+Shift+H` announces HP for all heroes in the party.
- `Ctrl+F` is used by `CombatHandler` to announce aura/curse effects for the focused combat character, or the active-turn character if no character is focused. Plain `F` remains deck/draw-pile access outside this modified combat chord.
- `Ctrl+I` is used by `CombatHandler` in combat to announce revealed enemy intention cards from Sight/Obzor. Plain `I` remains character information access.
- `Ctrl+M` is used by `ModSettingsHandler` to open the AccessTheObelisk settings menu. This replaces the previous all-currencies hotkey; individual currency hotkeys remain `Ctrl+G`, `Ctrl+D`, and `Ctrl+S`.
- `Ctrl+R` is used by `CombatHandler` to announce the current combat round.
- `/` is used by `ChatHandler` in multiplayer to open/focus the native chat input. `[` and `]` read older/newer visible chat history lines without taking persistent focus. While the native chat input is focused, pressing any arrow key deactivates the input so the same arrow can return to ordinary screen navigation.
- `P` is used by `PlayersPopupHandler` in multiplayer to open the native players popup with ping and mute/unmute controls. Plain `P` was not found in `InputController` key handling or the generated `ActionsController` keyboard bindings.
- `Ctrl+Shift+F9` toggles AccessTheObelisk debug logging and memory diagnostics. When enabled, the mod writes `[MEMORY]` lines to the BepInEx log immediately and every 30 seconds, including managed heap, process working set, process private bytes, current accessibility state, and active scene.
- `Home` and `End` are used by accessibility handlers to jump to the first or last item in the active mod-managed list/menu. `Ctrl+Home` and `Ctrl+End` jump to the first or last line in active detail/readback buffers, including the global event buffer when it is focused.

Avoid as mod single keys:

- Arrow keys, WASD, Enter, Escape, Tab, Space.
- Digits 0-9 and numpad digits.
- R, E, S, A, W, Q. In combat with keyboard shortcuts enabled, R/E/S/A/W/Q call `MatchManager.KeyboardEmote(0..5)`, so E is already reserved by the game.
- Ctrl or Alt by themselves.
- PageUp and PageDown.
- F1-F12, period, comma.

## UI System

Core patterns found:

- Most UI text is accessible as public `TMP_Text` fields on manager/component classes.
- Many buttons are represented by `Transform` entries in per-screen controller lists.
- Controller navigation moves the actual mouse cursor to the selected `Transform` with `Mouse.current.WarpCursorPosition()`.
- Hover state is often visual/audio only, so the mod should announce when the selected transform changes.

Important classes:

- `MainMenuManager`: main menu UI controller. Has `menuOps`, `profileOps`, `menuSaveButtons`, text fields such as `saveTitle`, `saveDescription`, `challengeTitle`, `challengeDescription`, and `profileMenuText`.
- Main menu multiplayer shortcut: `MainMenuManager.Multiplayer()` opens `ShowGameModeSelection(1)`, which shows the normal multiplayer game-mode choices and also activates `joinT`. `MainMenuManager.JoinMultiplayer()` sets Adventure as the game type and loads the `Lobby` scene immediately, without selecting a game mode or save slot first. Accessibility should expose `joinT` alongside the multiplayer game-mode tiles.
- Main menu profile selection: `MainMenuManager.ShowProfiles()` opens the native profile screen, fills `profileOpsText` from `SaveManager.GetProfileNames()`, hides `menuT`, shows `profilesT`, and shows `profileDelete` only when the current profile is not the default profile and has a name. `UseProfile(int)` selects an existing profile, closes the profile screen when selecting the current profile, or opens `AlertManager.AlertInput(...)` to create an empty non-default profile. `DeleteProfile()` opens the game's normal double-confirmation alert before deleting through `SaveManager.DeleteProfileFolder(...)`. Accessibility should use these existing methods and read visible `profileOpsText`/`profileDeleteText`. Do not treat `profilesT.activeSelf` alone as proof that the profile screen is open; require the profile transform to be visible, the main `menuT` to be hidden, and visible profile option transforms, otherwise startup can be misclassified as the profile screen.
- Main menu Paradox account widget: `MainMenuManager.ShowPDXPreLogin()`, `ShowPDXLogin()`, `ShowPDXCreate()`, and `ShowPDXLogged()` swap the visible Paradox popup under `paradoxT` after `ResetPDXScreens()`. Login uses `paradoxLoginUser`, `paradoxLoginPassword`, and `PDXLogin()`. Account creation uses `paradoxCreateEmail`, `paradoxCreatePassword`, `paradoxDropdownRegion`, `paradoxDropdownDay`, `paradoxDropdownMonth`, `paradoxDropdownYear`, `paradoxCreateOffers`, and `CreatePDXAccount()`. Other native actions include `LogoutPDX()`, `LinkSteamPDX()`, and `ShowPDXDocumentFromForm(int)` for the linked legal documents. The game's own `ControllerMovement(...)` always appends visible `menuControllerPDX` entries to the ordinary main-menu controller list, so accessibility should keep Paradox controls in the main-menu focus list rather than treating the pre-login widget as a separate screen. For reliability, Enter on the visible create/login submit buttons can call `ShowPDXCreate()`, `CreatePDXAccount()`, or `PDXLogin()` directly instead of relying only on raycast/button state.
- `MenuButton`: simple main-menu button component with public `TMP_Text buttonText`.
- `BotonMenuGameMode`: game mode tile used after choosing Play/New Game. It is not a normal Unity UI `Button`; it uses `optionText`, a `BoxCollider2D`, and `OnMouseUp()` to call `MainMenuManager.ShowSaveGame(...)`.
- `MenuSaveButton`: save slot tile used after choosing a game mode. Important public fields include `slotText`, `typeText`, `descriptionText`, `playersText`, `madnessText`, `versionText`, and `deleteButton`. `SelectThis()` starts a new run from an empty slot or loads an existing save. `DeleteThis()` opens the game's normal double-confirmation alert and, if accepted, deletes the slot through `SaveManager.DeleteGame(...)`.
- Paradox legal documents: `Paradox.Startup.ShowDocumentAction(Document doc, bool isFromStartup)` writes `doc.Text` to `MainMenuManager.Instance.paradoxDocumentText` and calls `MainMenuManager.ShowPDXDocument()`. The visible popup is `MainMenuManager.paradoxDocumentPopup`, scrolling uses `paradoxDocumentScrollRect`, and the close action is `MainMenuManager.ClosePDXDocument()`. While startup legal documents are pending, `ClosePDXDocument()` advances to the next document through `Startup.ShowDocumentFromStartup()`; after the last document it hides the document popup.
- `BotonGeneric`: common button component with public `TMP_Text text`, `idTranslate`, `idPopup`, `SetText(string)`, and `GetText()`.
- `SettingsManager`: global settings modal. Useful public members include `IsActive()`, `ShowSettings(bool)`, `SelectTab(int)`, `graphicsTab`, `audioTab`, `gameplayTab`, toggles such as `vsyncToggle`, `fullscreenToggle`, `extendedDescriptionsToggle`, `backgroundMuteToggle`, `fastModeToggle`, `keyboardShortcutsToggle`, sliders `masterVolumeSlider`, `effectsVolumeSlider`, `bsoVolumeSlider`, `ambienceVolumeSlider`, dropdowns `resolutionDropdown` and `languageDropdown`, and setter methods such as `SetResolution(int)`, `SetLanguage(int)`, `SetFullscreen(bool)`, `SetVsync(bool)`, `SetMasterVolume(float)`, and the gameplay/audio toggle setters. `ControllerMovement(...)` is empty in the decompiled source, so accessibility must provide its own keyboard focus model.
- `AlertManager`: modal alerts with public `TMP_Text alertText`, button text fields, and input fields.
- `PopupManager` and `Popup`: hover/tooltips; text is stored in private `TMP_Text` fields found from child paths.
- `CardItem`: cards have private TMP fields for title, description, target, requirement, energy, and type; these are initialized from child transforms and may need helper methods or reflection if direct data is unavailable.
- Card play requirements: `CardData.EffectRequired` stores effect prerequisites such as `stealth`, `stanzai`, and `stanzaii`. `CardData.GetRequireText()` returns the same localized requirement text the game writes to `CardItem.requireTextTM`, using `Texts.Instance.GetText("requireSkill")` plus `Globals.Instance.GetKeyNotesData(effectRequired).KeynoteName`; Stanza I/II requirements append `+`. Combat play validation in `MatchManager` has special cases: `stealth` can also be satisfied by `Stealthbonus`, `stanzai` can be satisfied by Stanza I/II/III, and `stanzaii` can be satisfied by Stanza II/III. Accessibility card speech should read `GetRequireText()` instead of duplicating those names.
- `TomeManager`: Tome of Knowledge manager. It is a persistent singleton and is shown through `TomeManager.Instance.ShowTome(true)`. Native entry points include `DoTomeMain()`, `DoTomeCards()`, `DoTomeItems()`, `DoTomeGlossary()`, `DoTomeRuns()`, `DoTomeScoreboard()`, `SelectTomeCards(int)`, `SelectTomeScores(int)`, `SetPage(int, bool)`, `DoNextPage()`, `DoPrevPage()`, `Search(string)`, `ResetSearch()`, `DoRun(int)`, `RunDetailClose()`, `RunDetailOpenCharacter(int)`, `RunDetailButton(int)`, `PrevWeekly()`, `NextWeekly()`, `PrevPath()`, and `NextPath()`. Useful public UI members include section transforms, section button transforms, `TomeButtons`, `cardContainer`, `Paginator`, `searchInput`, `scoresName`, `scoreButtons`, `buttonPrevWeekly`, `buttonNextWeekly`, `runsContainer`, `runDetails`, and `glossaryTexts`.
- `TomeButton`: category/filter button in the Tome. Public `tomeClass` maps to card classes, item slots, scoreboard filters, run detail subpages, and index buttons. `TomeHandler` activates these by calling the same `TomeManager` methods as `TomeButton.OnMouseUp()`.
- `TomeRun`: run-history entry in the Tome. Public text fields include `madnessText`, `description`, `date`, `type`, and `score`; clicking normally calls `TomeManager.DoRun(runIndex)`. The runtime GameObject name is `run_INDEX`, so the accessibility handler parses the index for activation.
- `TomeRunDetails`: run detail panel. Public fields expose the visible summary text, character buttons, path controls, item icons, traits, `cardListContainer`, and close button. Character deck entries are `CardVertical` children.
- `Score`: Tome scoreboard row. Public TMP fields `index`, `name`, and `score` contain the readable leaderboard row.
- `EventManager`: event screen has public text fields for `title`, `description`, `descriptionRolls`, `result`, `resultOK`, `resultOKc`, `resultKO`, `resultKOc`, `characterT`, and `continueButton`. Event options are `Reply` components under the event screen.
- Multiplayer event timing: `EventManager` can become active and display its title/description before network synchronization has created visible `Reply` components. Accessibility must not announce an empty-option error during this startup interval; it should queue the first option only after a visible `Reply` or Continue control exists.
- `IntroNewGameManager`: story intro/interstitial screen used for zone introductions and lore transitions, not just new-game intro. Useful public members include `title`, `body`, `buttonContinue`, `SkipIntro()`, and background transforms such as `bgWolfWars` and `bgSahti`. `BotonGeneric.Clicked()` on the continue button calls `SkipIntro()` when the active scene is `IntroNewGame` and the button name is `Skip`.
- `Reply`: event option component. Useful public members include `replyText`, `replyRollText`, `probDice`, `probPopup`, `replyButtonBlocked`, `GetEventReplyData()`, `GetOptionIndex()`, and `SelectThisOption()`. When `GetEventReplyData().ReplyActionText == Enums.EventAction.CharacterName`, the game only shows the matching subclass portrait and quotes the reply text; the speaking hero name must be recovered by matching `RequiredClass` to `EventManager.Heroes`.
- `MatchManager`: combat screen exposes key text fields such as energy, round, deck/discard counters, trait info, and character window routing. Round display uses public `roundTM`; `SetRoundText()` writes `Texts.Instance.GetText("roundNumber")` with private `currentRound`, and public `GameRound()` returns the same current round number.
- `HeroSelectionManager`: screen loaded from empty save slots for normal new games. Controller navigation is already implemented in `ControllerMovement(...)`, which builds a mixed list from `menuController`, visible begin/ready/madness/sandbox/seed controls, hero boxes, random dice, player boxes, and available `HeroSelection` entries.
- `HeroSelection`: hero portrait entry. Useful public fields/methods include `nameTM`, `rankTM`, `blocked`, `DlcBlocked`, `HeroPicked`, `GetHeroClass()`, `GetHeroClassSecondary()`, and `OnClickedController()`. Its `Id` maps to `Globals.Instance.GetSubClassData(id)`, whose localized `CharacterDescription` and `CharacterDescriptionStrength` are loaded from the `class` localization table and can be read for hero-selection speech.
- Hero selection pre-run hero information: `HeroSelection.RightClick()` and `BotHeroChar.OnMouseUp()` with `isHeroStats` open the native `CharPopup` through `charPopup.Init(subClassData)` and `charPopup.Show()`. `CharPopup.ShowStats()` displays hero stats, traits, resistances, initial item, and starting cards. There is no `HeroSelectionManager.ShowDeck(...)` or `CharacterWindowUI` deck window on this pre-run screen; the available native deck-like view before starting a run is the initial-card area inside `CharPopup`.
- Hero selection trait tab: `HeroSelectionManager.TraitsTabUpdate(SubClassData)` fills nine `TraitRollOver` entries from `SubClassData.Trait0`, `Trait1A/B`, `Trait2A/B`, `Trait3A/B`, and `Trait4A/B`; therefore trait descriptions do exist at hero selection when the game's trait localization is available. Trait names/descriptions should be read through `Texts.Instance.GetText(trait.Id, "traits")` and `Texts.Instance.GetText(trait.Id + "_description", "traits")`, with `TraitData.TraitName`/`Description` only as fallbacks. Trait-card entries use the game's `traitAddCard` text with a localized card name.
- Hero selection pre-run perks: `BotHeroChar.OnMouseUp()` with `isHeroStats == false` opens `PerkTree.Instance.Show(subClassData.Id)`. Assigned party-slot heroes remain available through `HeroSelectionManager.GetBoxHeroFromIndex(slotId)`, so the accessibility handler can open the same perk tree for a filled party slot without changing hero assignment.
- Hero selection action controls: the native controller list is built from `HeroSelectionManager.menuController` plus visible Begin, Ready, Madness, Sandbox, seed, weekly modifiers, follow, and party-slot controls. Back/return-style buttons may appear as ordinary `BotonGeneric` entries in `menuController` or elsewhere under `allGO`; accessibility should expose visible `BotonGeneric` controls with names/translation ids such as MainMenu, Back, Return, Close, Exit, or Cancel in the action zone and activate them through `BotonGeneric.Clicked()`.
- Hero run level-up: `CharacterWindowUI` opens the level tab automatically when a hero is ready to level up. `DrawLevelButtons(...)` marks the two trait choices for the current level as active when `Hero.IsReadyForLevelUp()` is true and the hero is owned by the local player. `TraitLevel.OnMouseUp()` calls `AtOManager.HeroLevelUp(heroIndex, traitData.Id)` in town or map contexts; there is no separate native "level up" button outside selecting one active trait. `AtOManager.HeroLevelUp(...)` calls `Hero.AssignTrait(traitId)`, optionally adds trait cards, then calls `Hero.LevelUp()` only after the trait assignment succeeds. `Hero.LevelUp()` adds `HeroData.HeroSubClass.MaxHp[currentLevel]` to current/max HP when available, then increments `Level`. Trait cells with `TraitData.TraitCard` show/add the card instead of a normal trait description; the game previews the upgraded card for the current trait tier.
- `BoxSelection`: party slot entry. Useful public fields/methods include `dice`, `boxPlayer`, `playerOwner`, `GetId()`, and `GetOwner()`. `HeroSelectionManager.GetBoxHeroFromIndex(int)` returns the hero assigned to a slot.
- `CinematicManager`: intro/outro cinematic screen. Text is shown through public `TMP_Text textBottom` and `textMiddle`; `DoText(cinematic, index, position)` fills those fields from `Texts.Instance.GetText("cinematicIntro" + index)` or `Texts.Instance.GetText("cinematicOutro" + index)`, then hides both fields after 6 seconds. After the cinematic ends, `EndCinematicCo()` calls `GameManager.Instance.ChangeScene("Map")`.
- `MapManager`: map screen. Public helpers include `GetMapNodeDict()`, `CanTravelToThisNode(Node)`, `PlayerSelectedNode(Node)`, `ControllerMovement(...)`, `IsMaskActive()`, `nodeActive`, and `Conflict`. The controller list is private, so the mod rebuilds a safe focus list from the current node plus active travelable nodes.
- Map node data speech: `MapHandler` should stay aligned with the game's normal `PopupNode` hover information. It may read the map location name, accessibility route coordinates, current/available status, action type, boss/town markers, visible assigned event name when the popup would reveal it, visible combat enemies when the popup would show combat icons, and visible ground text. It must not reveal hidden event names, event tier/rarity, event option requirements/rewards, quest start/end markers, path requirements, combat tier, follow-up events, rift/corruption availability, internal ids, raw descriptions, or `NodeData.Description` authoring notes.
- Map verbosity setting: when AccessTheObelisk `MapDetailsEnabled` is off, `MapHandler` keeps the game's map navigation and route preview behavior but speaks only the location name and logical coordinates during Ctrl-arrow map exploration. If the current route preview presents multiple sibling/child choices, it appends only the generic localized branch word.
- Map localization note: map speech should prefer live `Texts.Instance.GetText(...)` lookups over cached object fields for node names (`nodes`), requirement names/descriptions (`requirements`), event names (`events`), card/item rewards (`cards`), monster names (`monsters`), and class names (`class`). This keeps external localization mods such as RussianTranslation reflected in screen-reader speech.
- `MatchManager`: combat screen. Useful public members include `GameStatus`, `MatchIsOver`, `GetTeamHero()`, `GetTeamNPC()`, `GetHeroActive()`, `GetNPCActive()`, `GetHeroEnergy()`, `GetHeroHand(int)`, `IsYourTurn()`, `CardItemActive`, `EnergySelector`, and `ControllerMovement(...)`.
- `UIEnergySelector`: combat energy assignment overlay for cards that spend extra energy. It is available through `MatchManager.EnergySelector`; useful public members include `IsActive()`, `textInstructions`, `textAssignEnergy`, `AssignEnergyMore()`, `AssignEnergyLess()`, `AssignEnergyAction()`, and `AssignEnergyFromOutside(int)`. `MatchManager.WaitingForCardEnergyAssignment` remains true until `AssignEnergyAction(...)` completes.
- `MatchManager` add/discard selectors: discard selection uses public `DiscardSelector`, `WaitingForDiscardAssignment`, `WaitingForLookDiscardWindow`, `CardsLeftForDiscard()`, `SelectCardToDiscard(CardItem)`, and `AssignDiscardAction()` via `UIDiscardSelector.Action()`. Add-card selection uses public `WaitingForAddcardAssignment`, `CardsLeftForAddcard()`, `SelectCardToAddcard(CardItem)`, and `AssignAddcardAction()` via private `addcardSelector`, read by the mod with Harmony `AccessTools.Field`.
- `UICombatDeath`: hero death overlay reachable through `MatchManager.DeathScreen`. Useful public members include `textCharDeath`, `textInstructions`, `button`, `IsActive()`, and `TurnOffFromButton()`.
- Combat defeat retry prompt: when all heroes are dead and restart-combat is enabled, `MatchManager.FinishCombatCo()` shows `AlertManager.AlertConfirmDouble(Texts.Instance.GetText("combatWantToRetry"))`, sets `AlertManager.buttonClickDelegate = WantToRetry`, calls `AlertManager.ShowReloadIcon()`, and moves the alert with `SetRestartPosition()`. Accepting the right button sets confirm answer true and reloads combat; left/cancel continues defeat resolution.
- `AlertManager`: modal alert buttons use `alertTextLeftButton`, `alertTextRightButton`, and `alertTextSingleButton`; `SetConfirmAnswer(bool)` activates double-button choices, and `CloseAlert(true)` activates a single-button alert. `AlertCopyPaste(...)` uses `alertTextCP` plus read-only-looking `alertInputCP` and must be closed with `HideAlert()` without invoking `buttonClickDelegate`, because the single Close button can otherwise fire a stale delegate. `AlertPasteCopy(...)` uses `alertTextCP` plus `alertInputPC`; Enter should call the active import delegate, while Escape/cancel should clear the delegate and hide the alert.
- `TownManager`: town hub. Useful public members include `buildingForge`, `buildingAltar`, `buildingChurch`, `buildingCart`, `buildingArmory`, `townReady`, `townUpgrades`, `treasureItems`, `treasureItemsCommunity`, `tutorialBanner`, `tutorialBannerText`, `ShowTownUpgrades(bool, bool)`, and `Ready()`.
- `TownBuilding`: building hotspot. Useful public members include `idTitle` and `idDescription`. Building activation routes to existing `AtOManager` methods: `DoCardCraft()`, `DoCardUpgrade()`, `DoCardHealer()`, `DoCardDivination()`, and `DoItemShop("")`.
- `TownUpgradeWindow`: town supply-upgrades window opened from the town hub through `TownManager.ShowTownUpgrades(true)`. Useful public members include `elements`, `botonSupply`, `requiredTM`, `exitButton`, `sellSupplyT`, `sellSupplyButton`, `supplySellQuantity`, `supplySellResult`, `sellSupplyTransforms`, `Show(bool)`, `IsActive()`, `ShowSellSupply(bool)`, `ModifySupplyQuantity(int)`, and `SellSupplyAction()`.
- `BotonSupply`: individual town supply-upgrade button. Useful public members include `column`, `row`, `supplyId`, `available`, and `BuySupply()`. The game builds IDs as `townUpgrade_column_row`; text is available through `Texts.Instance.GetText(supplyId)`. Buying normally shows the game's `townAssignWarning` confirmation, then `BuySupply()` calls `PlayerManager.PlayerBuySupply(supplyId)` and refreshes town upgrades.
- `PlayerManager` town supply helpers: `GetPlayerSupplyActual()`, `PlayerHaveSupply(string)`, `PointsRequiredForSupply(string)`, `SupplyRequiredForSupply(string)`, `TotalPointsSpentInSupplys()`, `PlayerBuySupply(string)`, `ResetTownUpgrade()`, `GainSupply(int)`, and `SpendSupply(int)`.
- Currency read helpers: `AtOManager.GetPlayerGold()` reads current gold, `AtOManager.GetPlayerDust()` reads current dust/shards, and `PlayerManager.GetPlayerSupplyActual()` reads current supply.
- `TreasureRun`: town reward entry. Useful public members include `bGeneric`, `qText`, `treasureId`, and `claimed`.
- `CardCraftManager`: card craft screens use `craftType == 0` for the Altar upgrade screen, `craftType == 1` for the Church/remove-card screen, `craftType == 2` for Magic Forge card crafting, `craftType == 3` for Zingarian cart divination, and `craftType == 4` for Armory item buying. Useful public members include `cardsOwner`, `cardListContainer`, `cardItemContainer`, `divinationButtonContainer`, `divinationWaitingContainer`, `divinationWaitingMsg`, `buttonL`, `buttonR`, `buttonRemove`, `botSaveLoad`, `exitCraftButton`, `heroIndex`, and `craftType`; visible Magic Forge and Armory entries are stored in private `craftCardItemDict`. `exitCraftButton` must be exposed by the accessibility handler, especially for corruption rewards that open altar/church/shop screens after the corruption continue button is hidden. On the altar screen, selecting a deck card through `CardVertical.fMouseUp()` calls `CardCraftManager.SelectCard(...)`, which fills private `costA` and `costB`, writes those costs to private `BotonGeneric` buttons `BG_Left` and `BG_Right` through `ButtonText(...)`, and disables the buttons when `CanBuy("A"|"B")` is false. `BuyUpgrade(...)` silently returns if `CanBuy(...)` is false; the game does not show a separate not-enough-dust alert for altar upgrades.
- Magic Forge saved decks: `CardCraftManager.ShowSaveLoad()` toggles `cardCraftSave`, `botSaveLoad`, and calls private `LoadDecks(...)`. Saved deck slots are `DeckSlot[] deckSlot`. Empty slots show `saveButton`, disable the slot collider, and call `CardCraftManager.SaveDeck(auxInt)` through `BotonGeneric` name `SaveSlot`. Filled slots enable the collider for `LoadDeck(slot)`, hide save, show `deleteButton`, and call `RemoveDeck(auxInt)` through `BotonGeneric` name `RemoveSlot`. Applying a previewed saved deck uses `botCraftingDeck`/`BotCraftingDeck` -> `CraftDeck()`. Text access patterns:

- Prefer public component fields and public methods such as `BotonGeneric.GetText()`.
- For generic Unity UI, use `GetComponentInChildren<TMP_Text>()` as a fallback.
- For card details, prefer `CardItem`'s `CardData` over scraping TMP text when possible.
- Use `Functions.StripTagsString()` or an equivalent mod helper to remove TMP rich-text tags before speech.

## Localization

- Game localization is handled by `Texts.Instance.GetText(id, type = "")`.
- Current language is stored in `Globals.Instance.CurrentLang`.
- `Globals.SetLang(int)` maps language indexes to `es`, `zh-CN`, `ko`, `jp`, `zh-TW`, `de`, `fr`, and `en`.
- Translation files are loaded from Unity resources under `Lang/{lang}/`.
- Mod localization should keep its own `Loc.Get()` system, but can read `Globals.Instance.CurrentLang` to choose language.
- AccessTheObelisk mod localization files live next to the deployed plugin at `BepInEx\plugins\AccessTheObelisk\Localization\`.
- File format is UTF-8 text with one `key=value` entry per line. `#` starts a comment. Placeholders such as `{0}` must be preserved in translations.
- `Loc.Get()` uses the current game language, currently recognizing `ru` and `en` directly. Missing keys fall back to English.
- Localization files are checked for changes at runtime, so text fixes can be tested by editing the file without rebuilding the DLL.
- Community translation mods may add extra language files such as `BepInEx\plugins\RussianTranslation\Lang\ru\ru.txt`; when they integrate with the game's `Texts` table, accessibility handlers should still prefer `Texts.Instance.GetText(...)` so speech follows the active game language and falls back naturally to the game's own localization.
- `TextCleaner.ToSpeech()` replaces TextMeshPro sprite tags such as `<sprite name=gold>` with `Texts.Instance.GetText("gold")` or mod-local `speech_sprite_*` fallbacks before stripping rich-text tags, so icon-only text remains speakable. Special game sprites that are not localized by the game, such as `cardrandom` and `heart`, should get mod-local `speech_sprite_*` entries. The event `Rest` option uses `<sprite name=heart>` after the reward percent, meaning health/HP, not the literal word "heart".
- `TextCleaner.ToSpeech()` also normalizes punctuation for screen-reader speech after rich-text cleanup: it removes spaces before punctuation and percent signs, collapses repeated punctuation, treats TMP `<br>`/`<br3>` line breaks as comma pauses, and changes mid-string sentence periods into commas while preserving final periods and decimal numbers. Prefer improving this shared cleanup layer over adding one-off punctuation fixes in handlers.
- `TextCleaner.ToSpeech()` is called very frequently during focus and combat updates. Keep it fast: shared regex patterns are static/compiled for rich-text cleanup, rich-text cleanup only runs when `<` markers are present, punctuation cleanup only runs when a cheap character scan detects punctuation issues, and punctuation normalization is a single manual character pass rather than a chain of regex replacements.
- `TextCleaner.ToSpeech()` caches repeated short inputs and clears that cache when `Globals.Instance.CurrentLang` changes. This is especially important in combat snapshots, where the same localized character and aura/curse names are read repeatedly while polling HP/effect deltas.
- Card descriptions can encode gameplay meaning only through TextMeshPro color tags from `CardData.ColorTextArray(...)`: damage `#B00A00`, heal `#1E650F`, aura `#263ABC`, curse `#720070`, and system/default `#5E3016`. `TextCleaner.ToSpeech()` restores the category for bare numeric or `X` color-only fragments, such as turning a curse-colored `2` in `cardsDispel` into `2 curses`, while leaving percentages, HP text, and fragments that already include a sprite unchanged.
- Aura/curse descriptions should use `GameText.AuraCurseDescription(effect, charges, character)` when current charges and character context are available. The game's popup path applies placeholder replacements such as `<ResistAux1>`, `<ExplodeAtStacks>`, `<ACOnExplode>`, and `<HealPerChargeOnExplode>` before showing descriptions; reading raw `AuraCurseData.Description` can produce missing numbers like `All resistances - %` or `Upon reaching charges`.
- `GameText.AuraCurseDescription(effect, charges, character)` is cached by language, effect id/name, charge count, and character id. This avoids repeatedly applying runtime aura/curse modifiers and placeholder replacements for the same visible effect during combat focus and card detail rebuilds.
- Card flavor text is loaded from `Lang/{lang}/{lang}_cardsfluff.txt` through the game's `fluff` translation type, but stored on `CardData.Fluff` during `Globals.CreateCardClones()`. Accessibility card/item detail speech should read `CardData.Fluff` rather than parsing external localization files directly.
- Weekly challenge modifiers are `ChallengeTrait` entries from `ChallengeData.Traits`. `MadnessManager.ShowMadness()` displays each localized name with `Texts.Instance.GetText(trait.Id)` and stores the game's localized explanation in the same text object's `PopupText.text` from `Texts.Instance.GetText(trait.Id + "desc")`. Accessibility should read that popup description together with the visible modifier name.
